# 0003 — Phase 2.5: pitch replication + ranged combat

- **Status:** accepted
- **Author(s):** SkyrimReLive contributors
- **Created:** 2026-04-16
- **Supersedes:** —

## Summary

Replicate aim pitch alongside the existing yaw, and extend the Phase 2.3
combat hook so bow shots and combat spells deal cross-player damage.
Ghosts visibly look up/down and can hit you with arrows or a fireball.

## Motivation

Phase 2.3b shipped melee combat: TESHitEvent → CombatEvent → server
validation → DamageApply → stagger. It works because melee carries a
weapon (TESObjectWEAP, with `GetReach()` and `GetAttackDamage()`) and
the engine fires TESHitEvent on contact.

What's missing:

1. **Pitch.** Ghosts only orient on yaw, so a remote player aiming at
   the sky looks identical to one aiming at your feet. Bow draws don't
   visibly tilt. Conversation feels static even before you fire a
   single arrow.
2. **Ranged hits.** TESHitEvent does fire for arrows and spells, but
   the server's range check (`weapon_reach + 50` slack) was sized for
   melee. A bow's `GetReach()` is small or zero — every ranged hit
   currently fails server validation and gets dropped silently.

Together, these two gaps gate the next plausible gameplay mode: a
sniper-and-mage party engaging from a distance.

## Design

### Scope

**In:**
- Pitch on PlayerInput / PlayerState (new field, not a Transform
  change — keeps Transform's struct layout v2-stable).
- Ghost SetAngle uses (pitch, 0, yaw) so remote actors visibly aim.
- TESHitEvent handling extends to ranged sources: detect bow / staff /
  spell and route through a ranged code path.
- Server: a separate validation policy for ranged. Skip the melee
  reach check; trust the TESHitEvent's existence as proxy for
  "engine actually computed an intersection."
- Damage extraction for ranged: bow uses `GetAttackDamage()` from
  the bow itself (server already clamps); spells use a fixed sentinel
  for v1 since spell magnitude lookup needs a bigger detour
  (MagicItem effects + magnitude scaling).

**Out (deferred to a later step):**
- Projectile *visual* replication (the actual arrow flight). Receivers
  see the stagger and HP drop, not an arrow in flight. Phase 3.
- Spell magnitude calc per-effect (simple sentinel for now).
- Charge time / draw strength for bows (treat as full-draw).
- Concentration spells (channeled flames, etc.).
- Shouts (Voice system, separate proposal).
- Mounted ranged.

### Wire-format delta

`PlayerInput` and `PlayerState` each gain one float:

```
table PlayerInput {
    ...                       // existing fields
    pitch: float = 0;         // radians; +down, -up (Skyrim convention)
}

table PlayerState {
    ...                       // existing fields
    pitch: float = 0;
}
```

Both are FlatBuffer table appends — no version bump (still wire v2).
Old plugins ignore the new field; new plugins see 0 from old peers.

`CombatEvent` gains an explicit attack-class enum so the server knows
which validation policy applies:

```
enum AttackClass : ubyte { Melee = 0, BowArrow = 1, Spell = 2 }

table CombatEvent {
    ...                       // existing fields
    attack_class: AttackClass = Melee;
}
```

Default is `Melee` so existing clients remain valid (ranged was never
shipped, so there's no behavior change for them).

### Plugin: capture pitch

`Net::send_player_input` already reads yaw via
`player->GetAngleZ()`. Add `player->GetAngleX()` and ship it.

### Plugin: render pitch on ghosts

`Ghost::tick_main_thread` currently does:

```cpp
g.actor->SetAngle({0.0F, 0.0F, render.yaw});
```

Becomes:

```cpp
g.actor->SetAngle({render.pitch, 0.0F, render.yaw});
```

Snapshot history grows the pitch field; `lerp_snap` linearly
interpolates pitch (no shortest-arc gymnastics — pitch is bounded
to ~[-π/2, π/2] and never wraps).

### Plugin: classify the hit

In `Combat::HitEventSink::ProcessEvent`, source-form classification
extends to:

| Source TESForm           | AttackClass | Reach         | Damage source            |
|--------------------------|-------------|---------------|--------------------------|
| TESObjectWEAP (melee)    | Melee       | weap.GetReach() | weap.GetAttackDamage() |
| TESObjectWEAP (bow/xbow) | BowArrow    | 0 (n/a)       | weap.GetAttackDamage()   |
| TESObjectWEAP (staff)    | Spell       | 0             | sentinel (e.g. 25)       |
| SpellItem / MagicItem    | Spell       | 0             | sentinel                 |
| Anything else            | Melee       | 0             | unarmed default          |

The `projectile` field on TESHitEvent is non-zero for arrows/bolts;
that's the secondary signal.

### Server: split validation by class

`handle_combat_event` currently does one range check based on
`weapon_reach + REACH_SLACK`. Extend:

```rust
match event.attack_class() {
    AttackClass::Melee => {
        if dist > event.weapon_reach() + REACH_SLACK { reject(); }
    }
    AttackClass::BowArrow | AttackClass::Spell => {
        // Engine already computed an intersection; trust the existence
        // of the TESHitEvent itself. Add a generous max-range cap (5000
        // units = ~one cell) just to bound griefing.
        if dist > MAX_RANGED_DIST { reject(); }
    }
}
```

Rate limit (5 hits/s) stays the same — bows and spells fire faster
than melee but 5/s is a comfortable upper bound for both.

### Receive path

No changes. `DamageApply` stays as-is; the stagger animation fires
on the target's client regardless of attack class.

### PvP toggle (host-controlled)

Friend co-op shouldn't let one player accidentally one-shot another
while testing a power attack. Phase 2.5 adds a server-side PvP gate
controlled by the host:

```toml
# server config (skyrim-relive-server.toml)
pvp_enabled = false   # default off — opt in deliberately
```

Server enforcement (in `handle_combat_event`, runs before any
validation):

```rust
if !self.config.pvp_enabled {
    info!(%peer, target_id, "CombatEvent dropped: PvP disabled");
    // Silently no-op. Attacker sees no stagger on the target — that's
    // their visible feedback. ServerNotice surfacing is a follow-up.
    return;
}
```

Why server-only:
- Authority lives on the server (rule H1). Clients can't be trusted to
  honor a "no PvP" flag — the gate must be enforced where damage is
  applied.
- No plugin changes needed for v1. Existing combat hook keeps shipping
  CombatEvents; server just drops them when the flag is off.
- Toggle is a config edit + server restart. Live toggle without restart
  is a follow-up (would need a console command on the *server*, or an
  authenticated admin RPC from a host's plugin instance).

Default is **off**. Reasoning: surprise-PvP is the worst possible
default for a friend-trust mod ("we were just exploring and Bjorn
killed me"). Hosts who want PvP enable it deliberately.

PvE between players (helping each other fight a dragon) is unaffected
— PvE damage is local-engine and never leaves a single client today.

## Alternatives considered

1. **Add pitch to `Transform`.** Cleaner conceptually, but `Transform`
   is a FlatBuffer *struct* (fixed layout) and changing it means a
   wire version bump from v2 → v3. Sibling field on the table avoids
   the bump and lets old/new clients coexist during rollout.

2. **Don't replicate pitch — derive from anim graph.** Possible for
   bow draws but breaks for "look up at the sky" idle aiming and for
   spells (no draw animation tells you where the caster is pointing).

3. **Server simulates the projectile.** Proper Source-engine-style
   server-side projectile sim is the long-term answer, but for Phase
   2.5 trusting TESHitEvent's existence is an order of magnitude
   simpler and good enough for friend-trust scope (Anti-cheat is
   deferred per project policy anyway).

4. **Per-effect spell magnitude.** Walking the SpellItem → effect list
   → MagicItem.GetCostliestEffectItem chain is doable but adds
   ~50 lines and depends on perks/skills the server doesn't track.
   Sentinel value is a deliberate placeholder that ships value now
   and can be improved without a wire change.

## Open questions

- **Bow draw strength:** treat all shots as full-draw, or read the
  bow draw graph variable? Initial ship uses full-draw (server clamps
  upward anyway). Revisit if it feels wrong in playtesting.
- **Spell sentinel value:** 25 damage is a guess. Could expose as a
  server config field (`spell_damage_default = 25`) so hosts can tune.
- **Projectile pre-flight cancel:** if the attacker disconnects mid-
  arrow, do we care? Probably not — the engine will GC the projectile
  on the attacker's side and the target never sees it.
- **PvP-disabled UX feedback:** when the server drops a CombatEvent
  because PvP is off, the attacker sees no stagger and may think the
  network is broken. A `ServerNotice { text: string }` message rendered
  as a console toast is the obvious fix; deferred to a follow-up so it
  can be reused for other server→client text (kick reasons, chat).

## Migration / rollout

Wire-additive, no version bump. Self-hosters re-deploy server + plugin
together as usual. Friends with the older plugin keep working in
melee-only mode (CombatEvent without `attack_class` defaults to
Melee → existing path).
